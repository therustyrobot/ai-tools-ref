## Warnings

### WR-01: `categorize_all()` silently drops repos omitted from a partial model response

**File:** `scripts/categorize.py:138–143`

**Issue:** The loop that processes model results iterates only over keys returned by the model (`for full_name, info in result.items()`). When the model returns valid JSON that simply omits one or more repos from the batch (a realistic partial-response scenario), those repos are never written to `cat_map`. `parse_with_retry` only produces an all-"Other" fallback when both API calls return *unparseable* JSON; it does not detect missing keys. The repos silently disappear from the output file.

The downstream generator recovers gracefully because `group_by_categories_hierarchical()` calls `cat_map.get(repo["full_name"], {})` with a default, placing uncategorised repos in "Other > Other". However, the `categories.json` file itself will be missing entries for those repos, which would break any future consumer that expects full coverage.

The test `test_missing_from_result_gets_other` has the correct docstring intent — "Repo not in model response gets Other/Other" — but does **not** assert that `owner/repo1` appears in the result at all (`result` only contains `owner/repo0`). The test therefore passes without catching this gap.

**Fix:**
```python
# scripts/categorize.py  ─  inside categorize_all(), after the for-loop on result.items()

        for full_name, info in result.items():
            cat_map[full_name] = {
                "category":    info.get("category", "Other"),
                "subcategory": info.get("subcategory", "Other"),
                "slug":        category_to_slug(info.get("category", "Other")),
            }
        # NEW: assign Other/Other to any batch member the model omitted
        for name in batch_names:
            if name not in cat_map:
                print(f"[WARN] Model omitted {name} from response; assigning Other.")
                cat_map[name] = {
                    "category":    "Other",
                    "subcategory": "Other",
                    "slug":        "other",
                }
```

Companion test fix — add to `test_missing_from_result_gets_other`:
```python
        # repo1 was omitted from model response — must be assigned Other by categorize_all
        self.assertIn("owner/repo1", result)
        self.assertEqual(result["owner/repo1"]["category"], "Other")
        self.assertEqual(result["owner/repo1"]["slug"], "other")
```

### IN-01: `render_card()` inserts `html_url` into href attributes without `html.escape()`

**File:** `scripts/generate.py:220, 238`

**Issue:** `repo['html_url']` is embedded directly in two `href="..."` attribute values:

```python
<a href="{repo['html_url']}" target="_blank" ...>
```

GitHub API URLs (`https://github.com/owner/repo`) will never contain `"` or `>` in practice, so the immediate risk is low. However, if `repos.json` were ever sourced from a different origin or manually edited, a `html_url` containing `"` would break out of the attribute and allow attribute injection into the rendered page.

**Fix:**
```python
safe_url = html.escape(repo.get("html_url", "#"), quote=True)
# then use safe_url in both href= occurrences in render_card()
```

### IN-02: Marquee text extraction double-processes HTML entities, corrupting names containing `&`

**File:** `scripts/generate.py:409–418`

**Issue:** `render_page()` builds `marquee_content` by parsing display names back out of the already-HTML-rendered `nav_html` string, then calls `.upper()` on them:

```python
for line in nav_html.split("\n"):
    if "<span>" in line and "opacity-60" not in line and "href" not in line:
        m = re.search(r"<span>([^<]+)</span>", line)
        if m:
            marquee_parts.append(m.group(1).strip().upper())
```

`nav_html` already contains HTML-escaped text (`html.escape()` was applied upstream). For a category name like `"AI & ML"`, the nav contains `AI &amp; ML`; after `.upper()`, the captured value becomes `AI &AMP; ML`. When this is inserted back into the page, `&AMP;` is not a valid HTML5 named character reference (character references are case-sensitive in HTML5) so browsers render it literally as `&AMP;` rather than `&`.

Affected category names: `"AI & ML"`, `"Self-Hosting & Homelab"`, `"Dev Tools & CLI"`, `"DevOps & Infra"`, `"Data & Analytics"`.

**Fix:** Extract the plain-text display names from `CATEGORY_META` / `LANG_META` directly instead of re-parsing rendered HTML:

```python
# Simpler and correct:
marquee_parts = []
for cat_name in hier_groups:   # or groups, depending on which path was taken
    slug = language_to_slug(cat_name)
    if slug in CATEGORY_META:
        display_name, _ = CATEGORY_META[slug]
    else:
        display_name, _, _ = LANG_META.get(slug, (cat_name, "📦", "#888888"))
    marquee_parts.append(display_name.upper())
```
