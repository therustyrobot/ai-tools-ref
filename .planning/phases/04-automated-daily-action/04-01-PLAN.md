<objective>
Create `.github/workflows/update-gallery.yml` — the GitHub Actions workflow that automates the full gallery update pipeline.

Purpose: Wire the three existing Python scripts (`fetch_stars.py` → `categorize.py` → `generate.py`) into a scheduled, self-contained CI job that runs daily at 06:00 UTC and can also be triggered manually. The workflow commits `docs/index.html` when content changes and skips the commit entirely when nothing has changed, preventing noise in git history and recursive self-triggering.

## Locked Decisions (from 04-CONTEXT.md)

| Decision | Value |
|----------|-------|
| D-01 | Separate "Install dependencies" step — `pip install requests` — visible as distinct step in Action log |
| D-02 | `requests` is the only PyPI dependency; all other imports are stdlib |
| D-03 | Plain `pip install requests` — no version pin, no `--upgrade` |
| D-04 | No `continue-on-error` on any step; whole job fails on any step failure |
| D-05 | No retry logic — next day's run handles transient failures |
| D-06 | GitHub default email notification is sufficient; no Slack/Issues alerts |
| D-07 | No `setup-python` step — rely on `ubuntu-latest` pre-installed Python (3.12 available) |

## Key Constraints from RESEARCH.md

- Use `python3` (not bare `python`) — shebang is ignored when invoked directly from `run:`
- Declare `permissions: contents: write` + `models: read` at workflow level — omitting `models: read` causes a silent 401 on the GitHub Models API call
- Job-level `env: GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}` propagates to all steps automatically
- `git diff --staged --quiet && echo "..." && exit 0` is the correct no-change guard — `exit 0` is critical to prevent the step from failing when there are no changes

## No-Change Guard Logic (DEPLOY-02)

`git diff --staged --quiet` exits **0** when staged area has no diff (nothing changed).
The `&&` chain fires `exit 0`, terminating the step cleanly — skipping commit and push.
If stars changed, `git diff` exits **1** → `&&` does NOT fire → execution falls through to `git commit` + `git push`.

  <action>
Create the directory `.github/workflows/` and write the workflow file with EXACTLY this content (verbatim — do not modify any field):

```yaml
name: Update Gallery

on:
  schedule:
    - cron: '0 6 * * *'
  workflow_dispatch:

permissions:
  contents: write
  models: read

jobs:
  update-gallery:
    runs-on: ubuntu-latest
    env:
      GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    steps:
      - name: Verify account
        run: echo "Running as $GITHUB_ACTOR"

      - name: Checkout
        uses: actions/checkout@v4

      - name: Install dependencies
        run: pip install requests

      - name: Fetch starred repos
        run: python3 scripts/fetch_stars.py

      - name: Categorize repos
        run: python3 scripts/categorize.py

      - name: Generate HTML
        run: python3 scripts/generate.py

      - name: Commit changes
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "41898282+github-actions[bot]@users.noreply.github.com"
          git add docs/index.html
          git diff --staged --quiet && echo "No changes to commit" && exit 0
          git commit -m "chore: regenerate stars gallery [skip ci]"
          git push
```

Implementation notes:
- `python3` is used (not `python`) per RESEARCH.md — shebang is ignored when scripts are invoked via `run:` (D-07)
- Separate "Install dependencies" step (D-01) — not inlined with a script call
- No version pin on `requests` (D-03)
- No `setup-python` step (D-07)
- Job-level `env:` block propagates `GITHUB_TOKEN` to all steps (including all three Python scripts that read `os.environ["GITHUB_TOKEN"]`)
- `git add docs/index.html` only — never `git add -A`; `_data/` files are ephemeral and gitignored (DEPLOY-01)
- `[skip ci]` in commit message prevents recursive workflow triggering (DEPLOY-03)
- `41898282+github-actions[bot]@users.noreply.github.com` is the canonical numeric-ID email for the Actions bot (DEPLOY-01)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → GitHub API | `GITHUB_TOKEN` crosses here; scoped to repo owner's account |
| GitHub Models API | Bearer token auth; `models: read` permission required |
| Runner filesystem → git push | Workflow writes `docs/index.html` and pushes; `contents: write` required |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Elevation of Privilege | `GITHUB_TOKEN` permissions | mitigate | Declare minimal permissions: `contents: write` + `models: read` only — no `id-token: write`, no `packages: write`, no admin scope |
| T-04-02 | Denial of Service | Recursive workflow trigger | mitigate | `[skip ci]` in auto-commit message (DEPLOY-03) prevents workflow from re-triggering on its own commit |
| T-04-03 | Information Disclosure | `GITHUB_TOKEN` in logs | accept | Actions automatically redacts `secrets.*` values in log output; low residual risk |
| T-04-04 | Spoofing | Wrong account's stars fetched | mitigate | ACTION-04: `echo "Running as $GITHUB_ACTOR"` at job start makes account mismatch immediately visible in run logs |
| T-04-05 | Tampering | Unauthorized `workflow_dispatch` trigger | accept | Only users with write access to the repo can trigger `workflow_dispatch`; standard GitHub repo access control is sufficient |
| T-04-06 | Information Disclosure | `_data/` intermediate files | mitigate | `git add docs/index.html` only (DEPLOY-01); `_data/` is gitignored so even `git add -A` would not stage it, but explicit single-file add is defense-in-depth |
</threat_model>
